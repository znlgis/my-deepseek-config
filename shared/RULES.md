# 全局操作规则（DeepSeek × 多工具共享）

> 借鉴 [my-opencode-deepseek-config](https://github.com/znlgis/my-opencode-deepseek-config)
> 的「Token 效率优先、双模型分工」理念，抽取出与具体工具无关的通用规则。
> 各工具目录下的记忆文件（`CLAUDE.md` / `QWEN.md` / `GEMINI.md` / `AGENTS.md` /
> `IFLOW.md` 等）会引用或内嵌本文件的核心内容，只在其上补充工具专属设置。

## 核心理念：DeepSeek 双模型分工

用 DeepSeek 的两档模型做「成本感知」的分工，用最小的上下文成本达到最好的开发效果：

| 档位 | 官方模型 ID       | 定位     | 用途                                     |
| ---- | ----------------- | -------- | ---------------------------------------- |
| Pro  | `deepseek-reasoner` | 强推理   | 复杂规划、架构、根因分析、代码审查、重型实现 |
| Flash| `deepseek-chat`     | 快速通用 | 探索、检索、简单编辑、通用问答、轻量任务   |

- **Flash 优先**：明确定义的搜索 / 查找 / 小改动优先用 `deepseek-chat`。
- **Pro 升级**：`deepseek-chat` 力不从心时，携带完整上下文升级到 `deepseek-reasoner`。
- **Pro 专注推理**：规划、架构、审查、复杂实现只走 `deepseek-reasoner`。

> **模型命名说明**：本仓库统一使用 DeepSeek 官方当前可用的
> `deepseek-chat`（非思考）与 `deepseek-reasoner`（思考）。
> 参考仓库中的 `deepseek-v4-pro` / `deepseek-v4-flash` 为其双模型分工的命名思路，
> 待官方发布对应型号后，只需在各工具配置里替换模型 ID 即可，其余分工逻辑不变。

## DeepSeek 接入端点

- **OpenAI 兼容**：`https://api.deepseek.com/v1`（供 Qwen Code、Gemini CLI、Codex、iFlow、Kimi CLI 等使用）
- **Anthropic 兼容**：`https://api.deepseek.com/anthropic`（供 Claude Code 使用）
- **API Key**：在 https://platform.deepseek.com 申请，形如 `sk-...`，通过环境变量注入，**切勿写入仓库**。

## 核心原则

1. **先辨意图再动手。** 区分字面请求与真实目标。「看一下 X」/「解释 X」不是「改 X」。用户没有明确要求实现时，不要直接改文件。
2. **做能完整解决问题的最小改动。** 不碰无关代码，完整正确优于宽泛聪明。
3. **先读后写。** 不猜代码行为——打开它，用真实文件验证假设。
4. **并行处理独立工作。** 把多个独立的读取 / 搜索 / 检索合并成一批，而非串行。
5. **尊重角色边界。** 只读角色（审查、探索、检索）不改文件，只以文本汇报。
6. **未经请求不创建文件。** 不主动创建 README、文档或任何新文件。
7. **按任务大小选模型。** 搜索 / 查找 / 简单编辑 / 文档 → Flash；推理 / 复杂决策 / 多步分析 / 实现 → Pro。边界情形先用 Flash，确实力不从心再升级到 Pro。

## 语言

用操作系统当前语言回复用户。zh-CN 系统用中文，en-US 系统用英文；除非用户明确要求，否则不强制英文。

## 约束

- **不引入新模型。** 仅使用 `deepseek-chat` 与 `deepseek-reasoner`。
- **不擅自新增依赖**，除非用户明确说明理由。
- **纯配置优先。** 优先改提示词 / 配置，而非引入新工具。

## 多步任务纪律

任何 ≥2 步的任务：先写有序待办清单；同一时刻只保留一个「进行中」；每完成一项立即标记完成（不批量）；范围变化时更新清单。

## 上下文与 Token 效率

- **引用路径，别贴整文件。** 指向 `src/app.ts:42`，不要把整个文件塞进提示词。
- **委派而非堆积。** 大文件交由子代理 / 探索读取，不要全部灌入主上下文。
- **并行独立读取。** 需要读 3+ 个互不依赖的文件时，一次性发起。
- **积极压缩。** 一条探索线索结论清晰后即压缩，从干净的上下文开始实现。
- **快变库先检索。** 具体或高频变动的库 / API 不要凭记忆写，先查官方文档。

## 何时提问 vs 直接执行

仅在「存在多种影响差异大的解释」或「关键上下文缺失」时提问。否则选最佳默认值，说明假设后继续。

## 反模式（阻断）

- 不建 `utils.ts` / `helpers.ts` / `service.ts` 之类的万能文件——用描述性文件名。
- 代码 / 注释中不用 emoji（除非用户明确要求）。
- 不用「simply」「obviously」「clearly」等 AI 填充词。
- 不留空 catch 块；不留被注释掉的死代码。
- `@ts-ignore` / `@ts-expect-error` 必须附解释性注释。
- 未经请求不创建文件。

## 质量基线

- 匹配项目既有风格、命名与约定。
- 注释解释 WHY 而非 WHAT，且只在代码库已有注释处补充。
- 验证改动可构建 / 通过既有检查，不破坏调用方。
- 汇报发现时引用具体位置（`file:line`）。
- 交付前从头到尾复读每个改动文件，自我质疑后再输出。

## 代码风格（实现时）

- 优先 `const` 而非 `let`；用三元 / 提前返回替代重赋值。
- 能不用 `else` 就不用——用提前返回压平结构。
- 数据变换优先函数式（`map` / `filter` / `flatMap`）而非命令式 `for`。
- 只用一次的值就地内联，减少命名变量。
- 不用通配导入 / 无必要的导入别名。
